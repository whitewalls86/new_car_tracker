A production data pipeline for tracking car prices

CarTracker started as a personal project — I was car shopping and wanted a data-driven way to spot a good deal before it disappeared. What began as a simple scraper grew into a full production pipeline: continuous collection across 50+ make/model pairs, a lakehouse architecture across Postgres, MinIO, and DuckDB, and a deal-score model that turns 50M+ raw observations into a single number. It's as much a learning project as a portfolio piece — every architectural decision in here was made, reconsidered, and sometimes replaced as the system grew into something that actually had to work.

How data flows

Ingest
Scraper → MinIO

Raw HTML artifacts written to MinIO object storage, partitioned by date and artifact type.

Bronze
Processing → Postgres

Artifacts claimed from the queue, parsed, and written to staging event tables in Postgres.

Silver
Archiver → Parquet

Archiver bulk-flushes staging events to Parquet files on MinIO, partitioned by source and date.

Mart
dbt + DuckDB

dbt reads Parquet from MinIO and Postgres exports, producing silver and mart layers in DuckDB.

Serving
Dashboard

Streamlit queries the DuckDB mart layer directly for price history, deal scores, and trends.

Architecture at a glance

Click any card to learn why it was built that way.

Scraper FastAPI Why?

Continuously surveys Cars.com across 40+ make/model pairs — capturing market-level search results every 15 minutes and drilling into individual listings for full vehicle detail.

  • Two-pass collection strategy: broad market sweeps surface new listings and price changes; targeted detail scrapes enrich each listing with full specs, trim, dealer, and history
  • Raw HTML artifacts stored to MinIO on every pass — the source of truth is always replayable, not dependent on what the parser could extract at the time
  • Anti-detection layer (Chrome TLS fingerprinting + Cloudflare bypass) keeps the collection running continuously without manual intervention
  • Adaptive backoff system tracks block events per listing and backs off exponentially — scraper recovers on its own without operator involvement
  • Process-wide session cache avoids redundant auth handshakes across concurrent scrape workers
Processing FastAPI Why?

Transforms raw HTML artifacts into structured price observations — separating the data the pipeline needs right now from the data that feeds long-term analytics.

  • Every artifact is parsed from the original stored HTML — if the parser improves, historical data can be reprocessed from source without re-scraping
  • HOT tables (ops.*) hold current state for pipeline operations — fast, minimal, always up to date
  • Staging event tables buffer every observation until the archiver flushes them to Parquet — the silver layer is built from this event stream, not from HOT table snapshots
  • Idempotent by VIN: the same listing can be processed multiple times without producing duplicate records, so retries and replays are safe
Ops FastAPI Why?

The control plane for the stack — handles auth, configuration, and deploy coordination so every other service stays stateless and focused on its own job.

  • Caddy calls GET /auth/check on every protected request — Ops validates the Google OAuth token and DB role so no other service needs to know about auth at all
  • Search configs, tracked make/model pairs, and user roles all live here — one place to change pipeline behavior without touching service code
  • Deploy intent state machine signals all services to drain before a container restart — zero-downtime deploys without an external orchestrator
  • Claim lifecycle management prevents duplicate scrape work across concurrent workers — Ops owns the coordination, scraper and processing just ask permission
Archiver FastAPI Why?

Bridges the operational and analytical data systems — draining Postgres staging tables into partitioned Parquet on MinIO so each system stays focused on what it does best.

  • Postgres stays lean and fast because it only holds what the pipeline needs right now — the Archiver is what makes that possible by moving the permanent record out on a schedule
  • Staging tables are intentionally short-lived: written to during processing, flushed to Parquet by the Archiver, then truncated — Postgres never becomes the analytics store
  • Parquet files are hive-partitioned by source and date on MinIO — structured for the analytical queries that DuckDB and dbt will run downstream
  • Flush and truncate happen in a single transaction — if the write to MinIO fails, nothing is lost and the next run picks up where it left off
dbt Runner dbt + DuckDB Why?

The analytical layer — transforms the raw Parquet event stream into tested, documented data models that answer real questions about the market.

  • DuckDB reads Parquet directly from MinIO via the httpfs extension — no managed warehouse needed, the lake is the warehouse
  • Models are layered by responsibility: staging cleans and types the raw events, intermediate builds reusable assets like price history and national benchmarks, mart tables answer specific questions — deal scores, inventory coverage, scraper health
  • Deal score model combines five factors (MSRP discount, national price percentile, days on market, price drop history, inventory supply) into a single 0–100 score with tier labels — built in SQL so the logic is transparent and testable
  • dbt tests run on every pipeline execution — transformation correctness is enforced the same way application correctness is, not left to manual spot-checks
  • Exponential backoff logic for blocked listings lives entirely in a dbt model — keeping operational concerns queryable and auditable alongside the data they affect
Airflow Airflow Why?

Coordinates the full pipeline — scraping, processing, archiving, and analytics — across a dozen independent services that have no direct knowledge of each other.

  • Without an orchestration layer, each service would need to know what comes next — Airflow is the only thing that knows the full sequence, so the services themselves stay decoupled
  • Migrated from n8n: workflow logic in n8n lived in a UI, not in code — it couldn't be unit tested, version controlled, or reviewed like the rest of the codebase
  • Thin-DAG, fat-service pattern: DAGs contain only scheduling and HTTP calls, all business logic lives in the services — the testable surface area is in Python, not in a workflow config file
  • Twelve DAGs covering scrape rotation, artifact processing, staging flush, dbt builds, orphan claim expiry, queue cleanup, and deployment coordination
  • Deploy intent DAG coordinates zero-downtime restarts — Airflow doesn't just run the pipeline, it manages the pipeline's own lifecycle
Dashboard Streamlit Why?

The payoff of the pipeline — fast, recent market intelligence made possible by architectural decisions made at every layer upstream.

  • Sub-second query performance on 57M+ observations comes from the layers below: hive-partitioned Parquet, DuckDB's columnar engine, and pre-aggregated mart tables — the dashboard itself does almost no work
  • Data is recent, not live — the mart layer refreshes hourly, which is the right tradeoff for analytical queries that don't need to be real-time
  • Surfaces pipeline health alongside market data — scraper block rates, processing lag, and stale listings are visible here, turning operational issues into something you can see rather than hunt for in logs
  • Access-gated behind the same OAuth + DB role layer as the rest of the stack — the dashboard is a consumer of the architecture, not a special case in it
MinIO S3-compatible Why?

The storage backbone of the lakehouse — keeps the analytical record out of Postgres and in a format purpose-built for the queries that matter.

  • Postgres was never meant to hold years of append-only observation history — query performance degrades, storage costs grow, and the operational database starts doing two jobs badly instead of one well
  • Moving the historical record to Parquet on MinIO lets Postgres stay small and fast for pipeline operations while the analytical layer scales independently
  • Parquet's columnar format is why DuckDB can query 57M+ observations in under a second — row-oriented databases like Postgres simply can't match that for this access pattern
  • S3-compatible by design: MinIO is a right-sized choice for a single-server deployment, but the interface is identical to AWS S3 — the entire stack ports to cloud storage without a code change
  • Two tiers: raw GZIP HTML artifacts (bronze, 30-day retention) and Parquet observations (silver, indefinite) — retention policy matches the value of the data at each layer
Caddy Reverse Proxy Why?

The front door — handles TLS, Google OAuth, and role-based access so the rest of the stack is visible to the right people without any service needing to own auth logic.

  • This is a portfolio project that needs to be accessible to hiring managers and reviewers — Caddy makes that possible without bolting auth logic onto every service
  • forward_auth middleware intercepts every request and calls Ops to validate the Google OAuth token and DB role — auth is enforced at the edge, invisible to everything behind it
  • Single Caddyfile defines all routing: / → Dashboard, /airflow → Airflow, /grafana → Grafana, /info → public — the entire access model is one readable config file
  • Automatic TLS via Let's Encrypt — the site is production-grade even if the audience is small
Grafana Observability Why?

The answer to "how do we know if something's wrong?" — a single place to see the health of every service instead of hunting through container logs.

  • A pipeline this distributed fails silently without observability — a scraper block, a processing backlog, or a staging flush that stopped flushing won't throw an error you'll notice until the data goes stale
  • Prometheus scrapes metrics from every layer: Airflow DAG success rates, Postgres connection counts and query latency, MinIO throughput, and host-level resources — all in one place
  • Dashboards surface the questions that matter operationally: is the queue draining? Is the scraper getting blocked more than usual? Is the mart layer falling behind?
  • Observability was added as a first-class concern, not an afterthought — Prometheus scrape targets were provisioned via a dedicated Flyway migration alongside the services themselves
Testing & CI pytest · GitHub Actions Why?

A three-layer test strategy that matches the scope of the pipeline — because a project with this many moving pieces needs more than one kind of wrong covered.

  • A scraper, six services, twelve DAGs, fifteen dbt models, and a lakehouse backend all have different failure modes — a single test layer would miss most of them
  • Layer 1 — unit tests: fast, no database, cover parsing logic, backoff behavior, and service internals; run on every commit before anything else
  • Layer 2 — SQL smoke tests: run every critical query against the real Postgres schema before dbt executes — catch schema regressions that unit tests can't see
  • Layer 3 — integration tests: full request paths against a real database, real dbt builds against a seeded CI schema, real Airflow DAG execution — no mocks where real behavior matters
  • Schema changes are versioned Flyway migrations reviewed as PRs — ALTER TABLE in production without a migration history is its own category of wrong, and this project doesn't allow it
  • Enforced in CI on every commit: lint → unit → build → smoke → integration — a broken pipeline can't merge

Technical highlights

The Data Journey: 🥉 → 🥈 → 🥇

A car listing enters the pipeline as raw HTML and exits as a deal score. Between those two points it passes through three storage systems, each chosen for what it does best: MinIO holds the unprocessed artifact, Postgres buffers the parsed observation for pipeline operations, Parquet on MinIO preserves the permanent analytical record, and DuckDB transforms it into something a person can act on. No single tool does all of that well — the architecture exists because the data has different needs at different stages.

One database doing everything — and why that had to change

The first version stored everything in Postgres — observations, analytics, and operational state in the same database. Analytical queries slowed operational writes, and dbt ran against tables that were still being written to. The fix wasn't a bigger database — it was separating concerns: Postgres owns operational state only, staging tables act as short-lived write buffers, and the Archiver drains them to Parquet on a schedule. Each system now does one thing well.

The dependency nobody named

The pipeline had a hidden coupling. To know what to scrape next, it queried dbt — int_ tables that computed staleness scores, mapped listing IDs to VINs, and produced the scrape queue. That meant after every batch of scrapes, the pipeline needed a dbt rebuild before it could schedule the next one. A full build was taking 45 seconds. It couldn't run concurrently — a build lock had to be added after parallel runs started stepping on each other.

The carousel hints problem made it worse. Scraping vehicle A would sometimes surface price updates for B, C, and D in the same response. Those vehicles didn't need their own scrape — but the pipeline couldn't know that without running dbt first. Operational logic and analytical logic were tangled in the same models, running on the same schedule, blocking each other.

The question that reframed it

The fix came from asking a different question: what should dbt actually own?

dbt is good at one thing — transforming a stable, append-only record into analytical assets. Deal scores, price history, national benchmarks. That work runs hourly and can fail and retry without consequence. The scrape queue is different. It needs to reflect the current state of the world right now — which listings are stale, which VINs are blocked, which artifacts haven't been processed. That's operational state. Rebuilding a dbt model to answer "what do I do next?" is the wrong tool for the job.

The three-schema solution

Postgres gets an ops schema: HOT tables, one row per entity, always current state. The scrape queue becomes a plain Postgres view reading from those tables — no dbt, no rebuild cycle, no lock. Operational workers own these directly.

A staging schema holds event tables: one row per event, append-only. On a schedule, the Archiver drains them to hive-partitioned Parquet on MinIO and truncates. Postgres never becomes the analytics store — it hands the record off and stays lean.

dbt runs against the Parquet files via DuckDB's httpfs extension. The lake is the warehouse. Deal scores and price history are built from the permanent record, not from operational tables that are still being written to.

What changed

The dbt build that used to block the pipeline now runs independently, on its own schedule, without holding anything up. Operational queries that took seconds — building a scrape queue from a dbt model — now take milliseconds from a Postgres view of a HOT table. The staging tables give the analytical layer a clean event stream to work from, and when the Archiver flushes them, Postgres shrinks back down rather than growing.

MinIO started as a storage fix. It ended as the thing that made the separation possible — a clear boundary between what the pipeline needs right now and what the analytics layer needs eventually. Those two things have different access patterns, different failure modes, and different owners. Once the architecture reflected that, both got faster.

Why I replaced the workflow engine mid-project

n8n worked until the pipeline got complex enough that bugs in the orchestration layer were hard to find. The root problem: workflow logic lived in a visual UI, not in code — it couldn't be unit tested, version controlled, or reviewed like the rest of the codebase. A broken DAG looked the same as a passing one until data stopped arriving. The move to Airflow wasn't about features — it was about making the orchestration layer subject to the same standards as everything else. Thin DAGs, fat services: scheduling and HTTP calls live in Airflow, all business logic lives in Python where it can be tested, diffed, and trusted.

The problem with logic you can't test

The issue was that n8n wasn't just an orchestrator — it was holding business logic. SQL queries lived in Postgres nodes inside workflows. Conditional branching, error handling, retry decisions — all of it in a visual UI, exported as JSON. That meant none of it could be unit tested. None of it showed up in a diff in a meaningful way. When a workflow had a bug, diagnosing it meant clicking through a visual interface and guessing at which node was wrong.

The worst part: a broken workflow looked the same as a passing one. n8n would report success on a run that had silently dropped records or skipped a step it shouldn't have. The only way to catch those failures was to notice that data had stopped arriving — which could be hours later.

I was still patching n8n workflows on April 13th. Two days later, I started building Airflow.

The insight that drove the switch

The decision wasn't really about Airflow's feature set. It was about a principle that the n8n experience had made concrete: orchestration logic should be subject to the same standards as everything else.

If it can't be unit tested, it's not code — it's configuration. And configuration that holds business logic is the worst kind: it looks trustworthy because it's in the repo, but it isn't testable the way code is.

This became the fat server, thin DAG pattern. DAGs contain only two things: scheduling and HTTP calls. No business logic, no SQL, no conditional branching. All of that lives in the Python services — where it can be tested, diffed, and reviewed like any other function.

The migration

Airflow ran in parallel with n8n for four days while the DAGs were built and validated. On April 20th, the n8n workflows were deactivated. Nine days later, the container was removed from docker-compose entirely.

The cutover was clean because the services hadn't changed — only what was calling them. The scraper, processor, and archiver all exposed HTTP endpoints. n8n had been calling those endpoints. Airflow now calls those endpoints. Swapping the orchestrator didn't require touching the services.

What changed

A broken DAG now fails in CI before it ships. Every piece of business logic that used to live in a workflow node now lives in a Python function with a test. The twelve DAGs that run the pipeline today are readable in under a minute each — they're scheduling logic, nothing more.

The pipeline didn't get more capable when I switched to Airflow. It got more trustworthy. That turned out to matter more.

Deploying without breaking what's already running

Restarting a container mid-pipeline means lost data, orphaned claims, and partial writes that are hard to recover from. Every service exposes a /ready endpoint and tracks in-flight work with a shared job counter — when a deploy is signaled, Airflow sensors stop scheduling new runs and each service drains to idle before its container restarts. The coordination happens across services that have no direct knowledge of each other, mediated entirely through a single deploy_intent table in Postgres. No external orchestrator, no manual timing — the pipeline manages its own lifecycle.

50M+ observations, one server

The entire stack runs on a single VPS — one Postgres instance, one MinIO node, one Airflow scheduler, eleven Docker containers. That's a deliberate choice, not a limitation: the architecture is what keeps it manageable at this scale. Columnar Parquet storage means DuckDB doesn't break a sweat on 50M rows. Staging tables mean Postgres never holds more than it needs to. Thin DAGs mean Airflow isn't doing real work, just coordinating. The bottleneck at this scale isn't the hardware — it's whether the data model is right. This one is.
